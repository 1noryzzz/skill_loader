# Codex Skill 系统学习笔记（聚焦 Skill，不含 MCP）

> 目标：帮助不熟悉 Agent 系统设计的读者，理解成熟产品在 Skill 维度通常如何组织代码与运行流程。
> 说明：本文结合 `openai/codex` 的公开仓库结构进行工程化抽象，不展开 Rust 语法细节。

## 1. 先建立一个正确心智：Skill 不是“提示词片段”，而是“受治理的能力单元”

在小项目里，我们常把 skill 理解成一段 prompt 模板；但在成熟产品里，skill 更接近一个“能力对象”，通常包含：

- 元数据（名称、描述、作用域、是否启用）
- 触发规则（显式提及、隐式匹配、默认兜底）
- 注入策略（注入到哪一轮、注入哪些片段、注入优先级）
- 生命周期（发现、加载、缓存、失效、禁用）
- 运行治理（与配置、工作目录、安全策略联动）

因此，真正可扩展的 skill 系统，核心不是“匹配算法本身”，而是：

1. **发现与索引**（有哪些 skill）
2. **选择与路由**（用哪个 skill）
3. **注入与编排**（如何加入上下文）
4. **治理与可观测**（如何稳定运行）

---

## 2. 从代码结构看成熟实现：建议拆成 4 层

参考 Codex 仓库的组织方式，可抽象成如下分层：

### A. Skill 定义层（Definition）

职责：定义“skill 是什么”。

建议包含：

- `id` / `name` / `description`
- `content`（系统提示、任务模板、参考链接）
- `scope`（system/user/repo）
- `status`（enabled/disabled/deprecated）
- `version` / `updated_at`

**工程价值**：把“数据”与“路由逻辑”分离，后续更容易做迁移、灰度、审计。

### B. Skill 发现与索引层（Discovery & Index）

职责：在多个来源中发现 skill，并建立可快速查询的索引。

常见来源：

- 系统内置 skill（全局默认）
- 用户级 skill（个人目录）
- 仓库级 skill（项目私有）
- 管理员策略层（强制开关）

索引通常至少会支持：

- 按名称查找
- 按 alias / 显式提及符号查找（如 `$xxx`）
- 按路径/上下文做隐式匹配

### C. Skill 选择层（Selection）

职责：根据用户输入 + 上下文，从候选 skill 中选出“最终注入集”。

成熟实现一般不是单一规则，而是**优先级链路**：

1. **显式触发优先**：用户明确点名某 skill（可解释性最强）
2. **隐式匹配补充**：未显式点名时，根据输入语义/路径上下文选候选
3. **默认兜底**：都未命中时走基础 skill
4. **禁用/冲突裁决**：由策略层最终裁掉不允许的 skill

这套链路的优势是：

- 用户可控（explicit）
- 系统智能（implicit）
- 永不空转（default）
- 可治理（policy gate）

### D. Skill 注入层（Injection）

职责：把已选 skill 以正确顺序拼到最终 prompt 上下文。

典型关注点：

- 注入顺序（system > task > context）
- 去重与截断（避免 token 爆炸）
- 冲突处理（同类规则如何合并）
- debug 可追溯（记录“为什么注入了这个 skill”）

## 3. 成熟产品里最重要的“非算法能力”：治理与稳定性

很多人只关注“怎么匹配更准”，但产品化后更先出问题的是稳定性与可维护性。

建议你重点学习并尽早落地：

### 3.1 分层配置与覆盖顺序

当 system/user/repo 都能定义 skill 时，必须有明确覆盖规则，例如：

- repo 可覆盖 user 的默认项
- admin policy 可强制禁用某些 skill
- 同名 skill 版本冲突时取最新或高优先级来源

### 3.2 缓存与失效机制

成熟实现不会每轮都重新扫盘加载 skill，而是：

- 按配置快照和工作目录缓存“可用 skill 列表”
- 文件变化或配置变更时再失效重建

### 3.3 可观测性（Observability）

至少记录：

- 候选 skill 列表
- 最终选中 skill 列表
- 触发原因（显式/隐式/兜底）
- 被禁用原因（策略拦截）
- 注入 token 开销

没有这些指标，后续几乎无法调优。

## 4. 一个关键澄清：到底是“LLM 自己选 Skill”，还是“系统帮它选”？

你这个疑问非常关键。工程上通常不是二选一，而是**“协同决策”**：

- **系统（orchestrator）负责边界与治理**
  - 决定“有哪些 skill 可见”（来源、权限、禁用规则）
  - 决定“注入上限和冲突裁决”（防止上下文爆炸）
  - 决定“默认兜底行为”（避免模型无工具可用）
- **LLM 负责语义与任务推进**
  - 在可见集合内，理解用户真实意图
  - 触发显式提及（用户点名）或在多候选中推进执行

所以你看到 Codex 偏“系统先整理候选，再让模型在候选里工作”，并不代表“LLM 完全不参与选择”；它只是把高风险、不可控的部分（权限/治理/冲突）前置到系统层。

这和我们前面讨论并不矛盾：

1. 前面讲的 A/B/C 更多是“**候选生成与路由算法**”。
2. Codex 展现的是“**产品化治理骨架**”。
3. 实际成熟系统通常是两者叠加：先有治理骨架，再把 A/B/C 插到候选生成层。

> 一句话：不是“模型决定一切”，也不是“规则决定一切”，而是“系统收敛边界 + 模型做语义决策”。

## 5. 给你当前项目的渐进式路线（从简单到成熟）

你当前项目已经有“关键词规则 + 默认 skill”的基础，这是对的。下一步建议：

### Phase 1：规则工程化（现在就做）

- 把规则写成可配置表（而不是散落在代码里）
- 给每次选择输出 `selection_reason`
- 增加 `disabled_skills` 与 `forced_skills`

### Phase 2：候选召回（中期）

- 在现有规则后增加候选召回层（可先用简单关键词索引，再升级 embedding）
- 采用 `top_k + score threshold`
- 仍保留“显式触发优先”的硬规则

### Phase 3：策略治理（规模化）

- 增加多层来源（system/user/repo）
- 做缓存键（config hash + cwd）
- 完整记录决策链路，支持调试回放

> 这条路线能让你从当前小规模实现平滑升级，不会一次性引入复杂系统导致维护失控。

## 6. 一个可以直接复用的 Skill 选择流程（伪流程）

1. 收集当前会话可见 skill（system/user/repo）
2. 应用策略层过滤（禁用名单、权限、环境约束）
3. 解析用户显式提及 skill（最高优先级）
4. 对剩余问题做隐式匹配，得到候选 top_k
5. 合并显式与隐式结果并去重
6. 应用冲突裁决（同类型 skill 数量上限）
7. 如果为空，注入默认基础 skill
8. 记录 `decision_trace` 并输出注入结果

如果你未来想接入更复杂的 Router 或 RAG，这个骨架无需推翻，只需替换第 4 步的实现。

## 7. 学习时的“对照阅读法”建议

你不需要读懂 Rust 语法，也能通过目录理解架构：

- 先看 `skills` 相关目录：确认 skill 的“对象模型”与“manager”职责
- 再看 `injection`：确认“如何把 skill 放进上下文”
- 最后看配置结构：确认“禁用、覆盖、缓存”这些治理能力如何接入

阅读目标不是记住代码细节，而是回答 3 个问题：

1. 可用 skill 从哪里来？
2. 最终 skill 怎么被选中？
3. 选择结果如何可解释、可调试、可治理？

## 9. 一句话总结

成熟产品在 skill 侧的关键，不是“选 skill 的算法多高级”，而是**有没有一套可扩展、可治理、可观测的选择与注入骨架**。
